## 1. 초심자를 위한 비유 📦

### ✂️ 가위로 포장지 자르기 vs 🤖 자동 추출 로봇 (EL)
예전 방식의 JSP 스크립틀릿 표현식(`<%= %>`)을 사용할 때는 데이터 하나를 화면에 보여주기 위해 복잡한 자바 코드 호출과 형변환을 거쳐야 했습니다.
*   **스크립틀릿 방식**: `<%= request.getAttribute("name") %>`
    *   *상자를 부수고 알맹이를 손으로 일일이 꺼내는 것처럼 번거롭고 복잡합니다.*
*   **EL 방식**: `${name}`
    *   *상자에 붙은 "이름표"만 말하면 로봇이 알아서 자바 내부 객체를 가져와 주는 것처럼 직관적이고 단순합니다.*

---

### 🗄️ 데이터를 저장하는 3가지 상자 (Scope)
서블릿에서 데이터를 보관해두는 공간(Scope)은 보관 기간과 공유 대상에 따라 여러 종류가 있습니다.

1.  **당일 퀵배송 상자 (`Request`)**:
    *   나에게 배송되어 단 한 번 열어서 데이터를 전달해주고 나면 바로 파기되는 일회용 박스입니다. (단일 요청-응답 주기 동안만 유효)
    *   `req.setAttribute("name", "요루");`
2.  **개인용 사물함 (`Session`)**:
    *   사용자별로 주어지는 공간으로, 웹브라우저 창을 닫거나 로그아웃하기 전까지 지속적으로 유지됩니다. (사용자별 유지)
    *   `req.getSession().setAttribute("name", "아사");`
3.  **사무실 공동 탕비실 (`Application`)**:
    *   해당 웹 서버 안에서 일하는 모든 스레드(모든 유저)가 공동으로 참조하고 공유하는 영구적인 데이터 창고입니다. (서버 시작부터 종료 시까지 유지)
    *   `getServletContext().setAttribute("name", "에드워드");`

> 🙋‍♂️ **질문**: 도우미(EL)에게 `${name}`을 가져오라고 지시했는데, 3가지 상자 모두에 `name`이라는 데이터가 들어있다면 어떻게 되나요?
>
> 🤖 **답변**: 도우미는 **가장 가깝고 좁은 보관함부터 순서대로** 뒤져서 먼저 나온 데이터를 줍니다. 즉, `Request -> Session -> Application` 순서대로 탐색하여 `Request` 박스에 들어있던 `"요루"`가 출력되게 됩니다.

---

### 📝 규격 명함첩 (JavaBean) vs 최신 디지털 지갑 (Java Record)
*   **JavaBean ([Data.java](file:///Users/morgan/Documents/workspace/jsp/src/main/java/com/example/jsp/step3/Data.java))**: 전통적인 Java 규격에 따라 모든 데이터 필드에 `get`으로 시작하는 이름표(`getValue1()`, `getValue2()`)를 만들어 두는 방식입니다. EL은 `${data.value1}`을 부르면 자동으로 뒤에서 `getValue1()`을 연결해 줍니다.
*   **Java Record ([Data2.java](file:///Users/morgan/Documents/workspace/jsp/src/main/java/com/example/jsp/step3/Data2.java))**: 최신 자바 문법으로, `get`을 안 붙이고 변수명 그대로 메서드(`value1()`)를 만듭니다. 하지만 구버전 컴파일러나 기본 EL 도우미는 여전히 `get`이 붙은 명함만 찾을 수 있기 때문에, Record를 쓰더라도 수동으로 `getValue1()` 같은 다리를 놓아주어야 정상 작동합니다.

## 2. 능숙자를 위한 절차 설명 ⚙️

JSP 내에서 서블릿 데이터 스코프에 결합된 데이터 모델(JavaBean 및 Java Record)을 조회하고 처리하는 실무 절차입니다.

### Step 1: 서블릿 내 데이터 설정 및 영역 바인딩
[ElServlet.java](file:///Users/morgan/Documents/workspace/jsp/src/main/java/com/example/jsp/step3/ElServlet.java)처럼 비즈니스 요구사항에 적합한 데이터 바인딩 영역(Scope)을 정해 데이터를 주입합니다.

```java
// 1. request 영역 바인딩 (단일 요청 응답용)
req.setAttribute("data", new Data("문자열", 42, true));
req.setAttribute("data2", new Data2("공자왈맹자왈", 73, false));
req.setAttribute("name", "요루");

// 2. session 영역 바인딩 (사용자 고유 정보 유지)
HttpSession session = req.getSession();
session.setAttribute("name", "아사");

// 3. application 영역 바인딩 (서버 글로벌 상태 저장)
ServletContext context = getServletContext();
context.setAttribute("name", "에드워드");
```

### Step 2: JavaBean 및 Record 규격 관리
*   **JavaBean**: EL 파서가 리플렉션을 통해 Getter를 자동 감지하도록 표준 빈즈 네이밍 룰(`getPropertyName()`)에 맞춰 설계합니다.
*   **Java Record**: Record는 자바 컴파일 시 겟터가 필드명 그대로(예: `value1()`) 생성되므로, EL 사양에 따라 호환되도록 명시적인 Getter 메서드를 구현해야 합니다.

```java
// Data2.java (Java Record 예시)
public record Data2(String value1, int value2, boolean value3) {
    // EL 파서와의 하위 호환성을 위해 Bean 스타일의 Getter 재정의
    public String getValue1() { return value1; }
    public int getValue2() { return value2; }
    public boolean getValue3() { return value3; }
}
```

### Step 3: JSP에서 EL 표현식으로 렌더링
[el.jsp](file:///Users/morgan/Documents/workspace/jsp/src/main/webapp/WEB-INF/step03/el.jsp)에서 `${키값}` 형태로 자원을 출력합니다.

```jsp
<%-- 1. 단일 데이터 값 출력 --%>
<p>${name}은 황천의 츠가이 캐릭터 중 하나입니다</p>

<%-- 2. JavaBean의 프로퍼티 연계 --%>
<p>
    ${data.value1} <%-- 내부적으로 data.getValue1() 자동 실행 --%>
    ${data.value2}
    ${data.value3}
</p>

<%-- 3. Java Record 프로퍼티 연계 --%>
<p>
    ${data2.value1} <%-- 재정의한 getValue1() 실행 --%>
</p>
```

## 3. 취업 대비를 위한 면접 예상 문항 💬

### Q1. JSP EL(Expression Language) 문법에서 널(Null) 값이나 존재하지 않는 속성을 호출했을 때 어떻게 처리되나요? 기존 자바 스크립틀릿 표현식(`<%= %>`)과의 차이점도 함께 설명해 주세요.
> **답변 방향**: 예외 발생 여부(NPE vs 빈 문자열 출력) 및 프레젠테이션 레이어의 오류 내성에 관한 실무적 관점을 말합니다.

*   **JSP 스크립틀릿 표현식**: `<%= request.getAttribute("not_exist") %>`를 호출하고 그 객체의 내부 메서드를 호출하려고 하면 즉시 `NullPointerException`이 발생하여 서버 에러 화면(500 Status)이 사용자에게 노출됩니다.
*   **JSP EL**: `${not_exist}` 또는 `${not_exist.property}`를 실행하더라도 EL 파서는 내부적으로 `null` 검사를 수행하여 예외를 던지지 않고 **공백 문자(빈 문자열 `""`)로 대체**하여 웹 페이지를 안정적으로 렌더링합니다. 이로 인해 뷰 레이어의 안정성이 크게 제고됩니다.

### Q2. 여러 데이터 바인딩 스코프(Scope)에 동일한 key값으로 데이터가 세팅되어 있을 때, EL 문법(`${key}`)의 탐색 우선순위는 어떻게 되나요? 특정 영역을 직접 지정하고 싶다면 어떻게 해야 하나요?
> **답변 방향**: Scope의 범위(Page -> Request -> Session -> Application) 순서를 나열하고, EL 암시적 객체를 적용하는 실무적인 사용법을 묻는 질문입니다.

*   **탐색 우선순위**:
    1.  `pageScope` (가장 구체적이고 좁은 영역)
    2.  `requestScope`
    3.  `sessionScope`
    4.  `applicationScope` (가장 넓은 글로벌 영역)
*   **특정 영역 타겟팅**: 특정 범위의 데이터를 가리키고자 할 때는 EL 내장(암시적) 객체를 key 앞에 붙여 명시해 줍니다.
    *   예시: `sessionScope`에 바인딩된 `"name"`을 가지고 오려면 `${sessionScope.name}`으로 정의합니다.

### Q3. Java 14+의 최신 기능인 Record 모델의 데이터를 JSP EL로 출력하고자 할 때, 필드명과 매칭되지 않는 프로퍼티 오류(`PropertyNotFoundException`)가 나는 원인과 그 해결책을 서술하세요.
> **답변 방향**: 자바 레코드의 기본 컴파일 동작(Getter 네이밍 룰)과 서블릿 컨테이너(Tomcat/EL Engine)의 규격 미스매치를 인지하고 있는지 평가하는 고난도 질문입니다.

*   **원인**: JSP EL 파서는 기본적으로 JavaBean 스펙에 규정된 겟터 네이밍 표준(`get필드명()`, `is필드명()`)을 리플렉션 기술로 파싱하여 데이터를 꺼냅니다. 하지만 Java Record는 `get` 접두사를 배제하고 컴포넌트 명칭 그대로 메서드(`value1()`)를 자동 정의하므로, 구식 EL 스펙을 탑재한 서블릿 엔진은 이를 호출하는 데 실패하여 속성을 찾을 수 없다는 에러를 발생시킵니다.
*   **해결책**:
    1.  레코드 클래스 내에 `public String getValue1() { return value1; }`처럼 **자바 빈 규격을 준수하는 Getter 메서드를 수동으로 재정의**해 줍니다.
    2.  서블릿 런타임 환경을 Jakarta EE 10(Tomcat 10.1 이상, Java EE EL 5.0 사양 이상) 표준으로 업그레이드하면 EL 스펙 자체가 Record 클래스의 기본 접근자를 인식할 수 있으므로 별도 우회 코드 없이 정상 호출할 수 있습니다.
